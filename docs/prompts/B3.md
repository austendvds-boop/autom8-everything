# B3: Cross-Service Connectivity & Portal Checkout Fix

## Context
The portal checkout flow for Cadence provisioning calls cadence-v2 via `PORTAL_API_SECRET`. If the secret doesn't match, provisioning silently fails. The Cadence get-started wizard (`/cadence/get-started`) submits to `/api/onboarding/checkout` which was updated in B1 to use the portal checkout. This batch ensures the get-started wizard client-side code sends the right field names and adds cross-service validation.

## Tasks

### 1. Update CadenceGetStartedClient to use portal checkout fields
Read `src/app/cadence/get-started/CadenceGetStartedClient.tsx` and verify it sends the request to `/api/onboarding/checkout`. Update the request body to match the B1 route's expected shape:
- `business_name` (string)
- `owner_name` (string)  
- `owner_email` (string)
- `owner_phone` (string)
- `hours` (string)
- `business_description` (string)
- `faqs` (string)
- `area_code` (string, 3 digits)
- `transfer_number` (string)

If the wizard currently sends different field names (e.g., `businessName` instead of `business_name`), update the POST request to map correctly. The wizard should:
1. Collect the info in the wizard steps
2. POST to `/api/onboarding/checkout`
3. Receive `{ url: "https://checkout.stripe.com/..." }`
4. Redirect to that URL

### 2. Add Cadence connectivity test to health endpoint
Update `src/app/api/health/route.ts`:
- Add a `cadence` check that calls `${CADENCE_API_URL}/` and expects `{"status":"ok"}`
- Add a `cadencePortalAuth` check that calls `${CADENCE_API_URL}/api/portal/tenant/00000000-0000-0000-0000-000000000000` with the `X-Portal-Secret` header and expects a 404 (not 401). If it returns 401, the secret doesn't match.
- Wrap both in try/catch with 5s timeout using `AbortController`
- Report in the health JSON:
  ```json
  {
    "cadence": true,
    "cadencePortalAuth": true  // false if 401, true if 404 or 200
  }
  ```

### 3. Update CadenceWelcomeClient for portal flow
Read `src/app/cadence/welcome/CadenceWelcomeClient.tsx`. Currently it polls provisioning state. With the portal checkout flow:
- After Stripe checkout completes, the user lands at `/portal/checkout/success?session_id=...&email=...`
- The welcome page should either redirect there or be updated to explain that provisioning happens after payment

Check if the portal checkout success page (`src/app/portal/checkout/success/SuccessClient.tsx`) already handles the Cadence provisioning status display. If so, update the Cadence welcome page to redirect to the portal success page. If not, update the portal success page to show Cadence-specific messaging when `product=cadence`.

### 4. Ensure portal checkout success page handles Cadence
Read `src/app/portal/checkout/success/SuccessClient.tsx`. Make sure it:
- Shows appropriate copy for Cadence signups ("Your AI receptionist is being set up...")
- Shows appropriate copy for Review Funnel signups ("Your Review Funnel is active...")
- Has a link to the portal dashboard
